# v6.6 live-verification note (Constituent Alignment loyalty floor)

**Status: REQUIRED before treating v6.6 scores as trusted. Not runnable in
the cloud/PR environment — no production database exists there.**

v6.6 makes one behavioral change: Constituent Alignment's seat-relative vote
component no longer penalizes below-expected party loyalty — it floors at
neutral (50) instead of dropping toward 25. The change is deterministic and
fully unit-tested, so it needs **no calibration**. But one property is
inherent to any scoring change and can only be observed on the real scored
population: **flooring loyalists at 50 removes score spread that the old
penalty had manufactured, which could push the Constituent Alignment
(`score_independent_voting`) population standard deviation below its floor.**

Run the checks below after the **first full pipeline run under v6.6**
(`ALGORITHM_VERSION == "v6.6"`). Everything uses the same `docker exec
mp-backend-blue` pattern as `SCORE_AUDIT.md`; substitute the live container
name if different.

---

## Check 1 — Ground truth + distribution gate (the go/no-go)

This is the same gate the pipeline runs automatically; run it directly to see
the result for this change.

```bash
docker exec mp-backend-blue python3 - <<'EOF'
from app.database import SessionLocal
from app.pipeline.analyze.ground_truth import (
    check_ground_truth, check_score_distribution, MIN_STDEV,
)
import statistics
from app.models import Senator

db = SessionLocal()

gt = check_ground_truth(db)
print(f"Ground truth: {gt['checked'] - len(gt['failures'])}/{gt['checked']} passed")
for f in gt["failures"]:
    print(f"  X {f['senator']} {f['dimension']}={f['score']} outside {f['expected']} — {f['rationale']}")

dist = check_score_distribution(db)
iv = [s.score_independent_voting for s in db.query(Senator).all()
      if s.score_independent_voting is not None]
print(f"\nConstituent Alignment (IV): stdev={statistics.pstdev(iv):.2f} "
      f"floor={MIN_STDEV['score_independent_voting']} n={len(iv)}")
for f in dist:
    print(f"  X {f['dimension']}: {f['rationale']}")
db.close()
EOF
```

**Pass criteria**

- **Ossoff** lands in `[40, 60]` (his range was updated for v6.6). This is
  provable from the formula — a below-expected swing-state loyalist scores
  exactly 50 on the 80%-weighted seat-relative component, so final IV is 50
  with no coalition-breadth data, or `50*0.8 + breadth*0.2 ∈ [40,60]` with it.
  If he lands outside, the coalition-breadth term or his real break rate is
  not what was assumed — investigate before trusting the run.
- **Paul, Collins, Murkowski, Cruz, McConnell, Thune, Sanders, Warren,
  Klobuchar, Fetterman** are all above-expected crossers or safe-seat cases
  whose crossing side is unchanged in v6.6 — they should pass their existing
  ranges untouched. A failure here means something OTHER than v6.6 moved
  (data-fetch regression), not this change.
- **IV stdev ≥ 6.5** (the `MIN_STDEV` floor).

**If IV stdev < 6.5 — this is the one anticipated failure, and it does NOT
mean the change is wrong.** Flooring loyalists at 50 legitimately narrows the
lower half of the distribution because the prior spread there was
*manufactured by an unjustified penalty*, not real signal. The correct
response is to **lower `MIN_STDEV["score_independent_voting"]` with a
justifying comment** — exactly the call already made once when that floor went
`8.0 -> 6.5` (see the `MIN_STDEV` comment in
`backend/app/pipeline/analyze/ground_truth.py`). **Do NOT restore the loyalty
penalty to prop the stdev back up.** Before lowering, sanity-check it is not a
real collapse: confirm most senators still hold distinct IV values (a genuine
collapse looks like a large share sharing one near-identical value, the way
Promise Persistence did at ~3.4).

## Check 2 — Confirm the change did what it should, and only that

```bash
docker exec mp-backend-blue python3 - <<'EOF'
from app.database import SessionLocal
from app.models import Senator
from app.services.senator_service import get_senator_score_breakdown

db = SessionLocal()
for frag in ("Peters", "Ossoff"):
    s = db.query(Senator).filter(Senator.name.like(f"%{frag}%")).first()
    if not s:
        print(f"{frag}: not found"); continue
    bd = get_senator_score_breakdown(db, s.id)["independentVoting"]
    comp = bd["components"][0]  # seat-relative vote alignment
    print(f"{s.name} ({s.state}-{s.party}): IV={s.score_independent_voting} | "
          f"seat-relative={comp['score']}")
    print(f"   {comp['detail']}")
db.close()
EOF
```

**Expect:** for a below-expected loyalist (Peters is a D in D+1 Michigan
voting the party line), the seat-relative component reads exactly **50.0** and
the detail says *"more loyal than the seat expects, held at neutral (loyalty
is not penalized)."* If it reads below 50, the loyalty floor did not take
effect — confirm `ALGORITHM_VERSION == "v6.6"` actually deployed.

## Check 3 — Population-wide before/after (magnitude sanity)

After at least two v6.6-era runs exist, `SCORE_AUDIT.md` Step 8 (largest score
changes across the last two runs) shows the shift. Expect movement to be
**upward only, concentrated in swing/opposed-seat loyalists**, and no member
moved *down* by this change. Any downward IV move is not attributable to v6.6
and should be traced separately.

## Not shipped in v6.6 — do NOT enable without its own calibration run

Two improvements were deliberately withheld because they cannot be validated
without live data (see the "not shipped" note in
`backend/app/pipeline/analyze/score_calculator.py`):

1. **Member-flank crossing discount** — discounts above-expected crossing
   credit for members on their party's ideological wing (flank defectors like
   Paul/Sanders shouldn't be credited as moderates; Kirkland & Slapin 2017).
   Right in direction, but its magnitude changes real scores and must be fit
   against the live `ideology_score` distribution. **To land it:** re-add the
   discount + `_flank_extremity` + the `ideology_score` scoring param and its
   pipeline plumbing, then grid-search the constant against `GROUND_TRUTH` and
   the IV stdev floor on a real scored DB, set the constant AND the
   (then-known, not guessed) flank-affected ground-truth ranges, add unit
   tests, and only then ship. Do not ship a placeholder value.

2. **Positional-congruence model** — the faithful long-term construct (score
   the distance between a member's ideal point and their seat's expected
   position; Canes-Wrone 2002, Bafumi & Herron 2010) rather than a
   rate/direction proxy. Needs a party/coalition-relative target (every member
   is more extreme than their state's raw median) and its own calibration.
   Named as the metric's boundary, not started.
